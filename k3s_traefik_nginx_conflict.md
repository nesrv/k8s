# k3s (Traefik) и nginx на одном VPS: конфликт на портах 80/443

## Когда актуально

На одной машине уже работают **сайты за nginx** (Let’s Encrypt, виртуальные хосты), затем устанавливается **k3s** с встроенным **Traefik**. После этого часть или все HTTPS-сайты перестают открываться нормально, хотя **Gunicorn/Django** на `127.0.0.1` с заголовком `Host` могут отвечать **200**.

---

## Симптомы

- В браузере **404** или пустая/странная страница; иногда ответ **короткий** (`content-type: text/plain`, `content-length: 19`).
- Проверка сертификата показывает **не** Let’s Encrypt, а самоподписанный Traefik:

```bash
curl -sv -k --resolve ВАШ_ДОМЕН:443:127.0.0.1 https://ВАШ_ДОМЕН/ 2>&1 | grep -E 'subject:|issuer:'
```

Типично:

```text
subject: CN=TRAEFIK DEFAULT CERT
issuer: CN=TRAEFIK DEFAULT CERT
```

- При этом **`ss`/`lsof`** может показывать **nginx** на `:80` и `:443` — это не отменяет участия **Traefik** и **svclb-traefik** (балансировщик k3s для сервиса Traefik): трафик или сертификат всё равно может обрабатываться цепочкой k3s.

- В кластере есть поды:

```bash
sudo kubectl get pods -A | grep -i traefik
```

Ожидаемы **`traefik-…`** и **`svclb-traefik-…`** в статусе **Running**.

---

## Причина

По умолчанию k3s ставит **Traefik** как ingress-контроллер и поднимает инфраструктуру (**Service LoadBalancer**, **svclb**), которая взаимодействует с **хостовыми портами 80/443**. В результате HTTPS на публичном IP может завершаться **Traefik** (сертификат по умолчанию), а не **nginx** с вашими **ssl_certificate** для доменов. Маршрутов Ingress под ваши старые домены нет → **404** от дефолтного бэкенда Traefik.

Учебное приложение из методички с **NodePort** (например **30080**) **не требует** Traefik на 80/443.

---

## Решение (рекомендуется): отключить Traefik в k3s

### Вариант A — файл конфигурации

Создать или дополнить **`/etc/rancher/k3s/config.yaml`**:

```yaml
disable:
  - traefik
```

Перезапуск:

```bash
sudo systemctl restart k3s
```

Через 1–2 минуты проверить, что рабочие поды Traefik исчезли:

```bash
sudo kubectl get pods -A | grep -i traefik
```

Должны остаться в основном завершённые **Job** `helm-install-traefik-*` (их при желании можно потом удалить как мусор). Поды **`traefik-…`** и **`svclb-traefik-…`** в **Running** быть не должны.

### Вариант B — аргумент `server` в systemd

Если после рестарта Traefik снова появляется, проверьте полный юнит:

```bash
systemctl cat k3s --no-pager
```

В **`ExecStart`** после `k3s server` добавьте **`--disable traefik`** (с сохранением синтаксиса многострочной команды), затем:

```bash
sudo systemctl daemon-reload
sudo systemctl restart k3s
```

Документация: [K3s configuration / options](https://docs.k3s.io/installation/configuration).

---

## После отключения Traefik

1. Перезапустить nginx (если раньше не мог занять порты или были ошибки):

```bash
sudo nginx -t && sudo systemctl restart nginx
```

2. Проверить сертификат снова — ожидается **Let’s Encrypt** (или ваш выпуск) для домена, **не** `TRAEFIK DEFAULT CERT`:

```bash
curl -sv -k --resolve ПРИМЕР.ДОМЕН:443:127.0.0.1 https://ПРИМЕР.ДОМЕН/ 2>&1 | grep -E 'subject:|issuer:'
```

3. Убедиться, что приложение в кластере по-прежнему доступно по **NodePort**, например `http://IP:30080/docs`.

---

## Временная мера (без гарантии, что не восстановится)

Если нужно быстро проверить гипотезу до правки конфига:

```bash
sudo kubectl scale deployment traefik -n kube-system --replicas=0
sudo kubectl get ds -n kube-system   # имя DaemonSet svclb
sudo kubectl delete daemonset ИМЯ_svclb_traefik -n kube-system
```

Без **`disable: traefik`** в конфиге k3s компоненты могут снова создаться при обновлении или пересоздании манифестов — **постоянно** надёжнее **вариант A** или **B**.

## Связь с лабораторной работой

В **`laba_ci_cd_vps.md`** деплой в k3s идёт через **NodePort**; **Traefik на 80/443 не нужен**. Если на VPS уже есть **nginx** с сайтами, **отключение Traefik** — стандартный способ избежать конфликта с существующими виртуальными хостами и Let’s Encrypt.

---

## Зачем нужен Traefik (простыми словами)

В кластере у приложений **внутренние** адреса (Service: имя, порт внутри Kubernetes). С **интернета** люди заходят по **одному IP** и **разным доменам** (`shop.example.com`, `api.example.com` и т.д.).

**Traefik** — это **входная дверь** (ingress-контроллер): обычно слушает **80/443** на ноде (или за балансировщиком), по **имени хоста** и **пути** решает, в какой Service/Pod отправить запрос, часто оформляет **HTTPS** (сертификаты, в т.ч. Let’s Encrypt).

Без чего-то вроде Traefik (или **nginx Ingress**, **Cilium Gateway**, HAProxy и т.п.) **с доменом на 443** до пода извне не дойти «из коробки» — остаются **NodePort** (`http://IP:30080`), **LoadBalancer** (если настроен), либо **свой nginx на хосте**, который проксирует на порт кластера.

**Traefik как продукт не обязателен** — обязательна только **роль**: кто-то должен маршрутизировать внешний HTTP(S) на нужные сервисы. k3s по умолчанию ставит **именно Traefik**, чтобы эта роль была готова сразу после установки.

---

## Будет ли k3s без Traefik работать полноценно

**Да.** k3s — это **Kubernetes**; для работы кластера Traefik **не нужен**.

Без Traefik сохраняются:

- Pods, Deployments, ReplicaSet, **Service** (**ClusterIP**, **NodePort**);
- **kubectl**, API-сервер, **etcd** (в составе k3s);
- учебный сценарий с **NodePort** (например **30080**) — без изменений.

Не будет только **встроенного** ingress от k3s на **80/443**; при необходимости его заменяют другим Ingress-контроллером или оставляют **nginx на VPS** снаружи кластера.

**Кратко:** Traefik — удобная **дверь с доменами** внутрь кластера; **k3s без него полноценен**, просто вход «как у обычного сайта на 443» нужно обеспечить иначе (NodePort, хостовый nginx, другой Ingress).

## Краткая шпаргалка

| Признак | Интерпретация |
|--------|----------------|
| `CN=TRAEFIK DEFAULT CERT` в `curl` | TLS отдаёт Traefik, не ваш nginx LE. |
| Поды `traefik`, `svclb-traefik` Running | Traefik в кластере активен — отключить через конфиг k3s. |
| Gunicorn на `127.0.0.1:8002` + `Host` → 200 | Бэкенд сайта жив; чинить фронт (443), не Django. |
